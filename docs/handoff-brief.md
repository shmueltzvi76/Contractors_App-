## 0. איך לתקשר עם הבעלים (סגנון מחייב — israeli-tesla-standard)

הבעלים מצפה לסגנון ספציפי. הקפד עליו:

- **עברית מלאה** בתשובות (השאלה בעברית = התשובה בעברית).
- פורמט RTL: **כל רעיון בשורה נפרדת**, מונחים טכניים בשורה לעצמם בתוך \` \`, בלי לערבב עברית ואנגלית באותה שורה, בלי סימן פיסוק בסוף שורת עברית.

- הנדסה לפני קוד: לפרק לאקסיומות, להריץ יבש כל מקרה קצה לפני שורת קוד ראשונה, ארכיטקטורת אפס-קריסות, קוד נקי סופי בלבד — בלי גרסאות ביניים.
- **המטרה העל: שילד יוכל להבין את האפליקציה.** כל החלטת UX נמדדת בזה.

## 1. רקע ומטרה (Background & Purpose)

**מה זה:** כלי ניהול עסקי לבעלי מקצוע בשטח בישראל — חשמלאים, שיפוצניקים, קבלנים.
**מה הוא עושה:** הצעות מחיר מקצועיות, חשבוניות, מעקב תשלומים, חלוקת רווח לעובדים ושותפים, ניהול לקוחות (יומן עבודות), דשבורד רווחיות, ושילוב AI (Gemini) לבדיקת רווחיות וניסוח.
**קהל יעד:** קבלן עצמאי בשטח, מהטלפון, בעברית, RTL, mobile-first.
**כוונה מסחרית:** מוצר B2B בתשלום. זה לא פרויקט לימוד — היעד הוא שחרור מסחרי עם מאגר נתונים אמיתי ומנגנון מנוי.

## 3. ארכיטקטורה טכנית (Tech Architecture)

- **Frontend:** React (single-file, ~5000 שורות JSX).
- **עיצוב:** Tailwind CSS דרך Play CDN (אין שלב build). Dark mode מיושם בהזרקת CSS inline, **לא** דרך מחלקות `dark:` (כי אין build).
- **גופן/צבע:** Heebo, סכמה כחולה, mobile-first RTL.
- **Backend:** Firebase — Firestore (נתונים) + Auth.
- **AI:** Gemini Flash, מפתח per-user. **המפתח נשמר ב-localStorage בלבד, לעולם לא ב-Firestore** (עיקרון אבטחה מחייב — חל על כל מפתח רגיש).
- **דיפלוי/QA נוכחי:** Google AI Studio.
- **אייקונים:** `lucide-react` (מיובא כ-`* as Ic`). גרפים: `recharts`.

### האילוץ הקריטי ביותר
משתני Firebase הגלובליים `__firebase_config`, `__initial_auth_token`, `__app_id` **מוזרקים רק בתוך Google AI Studio בזמן ריצה**. מחוץ לו האפליקציה לא מתחברת ל-Firebase. לכן:
- בדיקת QA מלאה אפשרית כיום רק ב-Studio.
- בנינו **קובץ preview עצמאי** עם מוק של Firebase כדי לראות את ה-UI חי מחוץ ל-Studio (פירוט בסעיף 7).

## 4. אילוצים קריטיים (Critical Constraints)

1. **אין שלב build** — אסור להניח Tailwind compiler, אסור `dark:`, אסור JSX שדורש טרנספילציה מעבר ל-preset-react בסיסי.
2. **קובץ יחיד** — כל הקוד ב-`App_v43.jsx`. שמירה על המבנה הזה (אלא אם מחליטים על מיגרציה — סעיף 10).
3. **כל פיצ'ר חדש כבוי כברירת מחדל**, נפתח דרך ההגדרות.
4. **`geminiKey` לא נכתב ל-Firestore לעולם** (localStorage בלבד).
5. **מניעת רקורסיה** בחישובי רווח: עובד מסוג "אחוז רווח" לא נכלל בבסיס השכר ששימש לחישוב חלקו שלו.
6. **כל שינוי חייב לעבור compile-test** ובדיקות יבשות (dry-run) לפני שמועבר.

## 5. מודל הנתונים (Firestore Data Model)

נתיב בסיס: `artifacts/{appId}/users/{uid}/...`

| אוסף | מסמך | תוכן |
|---|---|---|
| `diary/{autoId}` | עבודה/הצעה/חשבונית | אובייקט עבודה (ראה למטה) |
| `workers/{id}` | עובד | `{id,name,phone,payType:'hour'|'fixed'|'profit',rate,fixedAmount,profitPct,dailyOverride,dailyAmount,dailyHours}` |
| `partners/{id}` | שותף | `{id,name,phone,payType:'pct'|'fixed',base:'profit'|...,pct,amount,always}` |
| `settings/profile` | פרופיל/הגדרות | כל אובייקט ההגדרות (ראה `buildSettingsPayload`) |
| `settings/draft` | טיוטת מחשבון | autosave של הטופס הנוכחי |

נתיב ציבורי: `artifacts/{appId}/public/data/announcements` (הכרזות).

**צורת רשומת `diary` (entry):**
```
{
  f: { n,p,city,street,proj, d, de(מקדמה/שולם), di(הנחה), dt('amount'|'percent'),
       ru(עיגול), q(האם הצעה), qm, rc, noteIn, noteOut, ms },
  j: [ job... ],            // מערך עבודות
  sub, dAmt, vatAmt, vat, vatRate, fin, qMax, am, ua, ca,
  st: 'pending'|'credit'|'completed'|'deleted',
  creditAmount, asg:[], revCount, pmHistory:[...], docNum?
}
```
**צורת `job` (blankJob):**
```
{ id, mode:'hour'|'qty'|'area', t(כותרת), d(תיאור), h(שעות), r(תעריף),
  m(חומרים), mm(אחוז התייקרות חומרים), tf, df, tr(נסיעה דק'), cw, v,
  qty, pu, qa, p1, p2, area, pm }
```

> ⚠️ **חשוב:** `fin` נדרס לאחר תשלום ל**יתרה הנותרת**, לא לסכום המקורי. הסכום המקורי משוחזר מ-`sub - dAmt + vatAmt`. כל תצוגה שמראה "סה"כ" חייבת לשחזר כך, לא לקרוא `fin` ישירות.

## 6. מנועי הליבה (Core Engines)

- **מנוע תמחור (`calcJ`/merge):** שלושה מצבים — שעה / כמות / מ"ר. חומרים עם אחוז התייקרות (`mm`). נסיעה עם סף "חינם" (`travelFree`, ברירת מחדל 30 דק'). רצפת מחיר = `settings.hr` (תעריף שעתי) — מוחלת **רק על עבודות שעתיות**. מע"מ לפי סוג עוסק. הנחה (סכום/אחוז). טווח הצעה (`qMax`).
- **ספר תשלומים (`pay`/`editPayment`/`removePayment`):** היסטוריה ב-`pmHistory`. `de` = סך ששולם. תשלום-יתר → סטטוס `credit` (לא בהצעות). הצעת מחיר לא נושאת זיכוי.
- **חלוקת רווח (`entryBases`/partners/workers):** בסיסים שונים (מחזור / אחרי בסיס / אחרי הכל). מניעת רקורסיה לעובדי-רווח (סעיף 4.5).
- **שמירת הגדרות (`buildSettingsPayload`):** שומר **כל** מפתח הגדרה דינמית פרט ל-`geminiKey` ו-`show`. מנקה `expenses`/`milestones`. **+ autosave** עם debounce, מוגן ב-`loaded` כדי שברירות מחדל לא ידרסו פרופיל שמור.

## 7. סביבת פיתוח ובדיקה (Dev & Test Workflow)

**זרימת העבודה שסיכמנו:**
1. המתקן מייצר תיקון בקובץ אחד נקי.
2. הבעלים שומר גרסה ממוספרת (App_vNN) כהיסטוריה.
3. הבעלים בודק ב-Studio (סביבה אמיתית).
4. דיווח באג = צילום מסך + תיאור.

**קובץ ה-preview העצמאי (לבדיקה מחוץ ל-Studio):**
- HTML יחיד עם React/lucide/recharts מ-`esm.sh` דרך importmap, Tailwind מ-Play CDN, ו-Babel standalone (module mode) לטרנספילציית JSX בדפדפן.
- **מוק Firebase in-memory** מחליף את `firebase/app|auth|firestore` עם אותו API בדיוק (initializeApp, getAuth, onSnapshot, setDoc, getDoc, getDocs, updateDoc, deleteDoc, writeBatch, query/orderBy/limit/startAfter). מאפשר לראות את כל ה-UI חי, בלי Firebase אמיתי. הנתונים בזיכרון בלבד (רענון מאפס).
- כך אפשר לראות חי בדפדפן/טלפון לפני העלאה ל-Studio. דיאלוגים (confirm/alert) עובדים מלא רק בקובץ שמורד ונפתח בדפדפן רגיל.

**ל-Claude Code:** עדיף להקים **פרויקט Firebase אמיתי** (סעיף 10) כדי לבדוק חי עם שמירה אמיתית, במקום המוק.

## 8. מה עשינו עד כה (היסטוריית הסשן — "הזיכרון")

נקודת התחלה: `App_v37`. גרסה נוכחית: `App_v43`.

**תיקוני לוגיקה מוכחים (compile + dry-run tests):**
- `NaN` בסכום חומרים: `Number(x.m)` בלי `||0` ב-10 מקומות → הרעיל דשבורד/דוחות/חישובי שותפים על דאטה מיובאת. תוקן.
- רצפת מחיר ניפחה עבודות כמות/מ"ר → הוגבלה לעבודות שעתיות בלבד.
- `pay()` חסם תשלום-יתר → עכשיו מאפשר ורושם זיכוי, עם אישור מפני טעות הקלדה.
- סף נסיעה-חינם לא עקבי (30 בתמחור מול 0 בבסיס שותף) → אוחד ל-30.

**שדרוג הגדרות:**
- שמירה דינמית (`buildSettingsPayload`) במקום אובייקט 100 שדות ידני — שום הגדרה לא "נופלת" יותר. `geminiKey` מוחרג.
- **Autosave** להגדרות, מוגן נגד דריסת פרופיל.
- 💡 (`HelpBtn`) עם הסבר+דוגמה ל-18 שדות קונפיגורציה (סה"כ ~81).
- חיפוש הגדרות מקיף (token-based) + מילות מפתח עשירות.
- שדות לא-חובה סומנו "(לא חובה)".
- בורר הסגנון (שעה/כמות) נחשף כבסיסי (לא רק Pro).
- גיבוי נחשף במצב פשוט.
- חלון ה-💡 תוקן: מתיבה ברוחב קבוע שנחתכה, לחלון תחתון ממורכז שלא נחתך ולא מזיז layout.
- טוגלי ₪/% קיבלו `title` + הסברי 💡.

**אומת שורה-שורה מול המקור:** המחיקה היחידה בכל ההיסטוריה היא שורת חסימת תשלום-היתר הישנה. שום שדה/פיצ'ר לא נמחק.

## 9. מה ידוע, לא טופל, או לא נבדק (Known Issues & Un-audited Areas)

**הבעלים צודק שנשארו באגים.** מה שלא נסקר/לא נבדק ויזואלית (כי אי אפשר היה להריץ React מלא בסביבת התיקון):

- כל זרימת ה-UI של היומן והדשבורד פיקסל-אחר-פיקסל (גלישות, יישור RTL, קפיצות layout בלחיצות).

- בלבול `hr` מול `hourFloor` (הושאר בכוונה — `hourFloor` רק סף אזהרה, אין לו שדה; כדאי החלטת מוצר).

- אימות אנונימי כיום (Studio) — לא מתאים לפרודקשן.

**משימה ל-Claude Code:** מעבר QA מלא חי, מסך-מסך, על כל הזרימות, עם הרצה אמיתית. כאן יתגלו רוב הבאגים שנותרו.

## 10. מפת דרכים למוצר מסחרי משלם (Roadmap to Production)

> **גיט מול Firebase — הבחנה חשובה (הבעלים שאל על זה):** שני דברים שונים ומשלימים, לא חלופות.
> `git` = היכן הקוד חי ומנוהל (היסטוריה, חזרה אחורה — מחליף את שמירת ה-App_vNN הידנית).
> `Firebase` = היכן הדאטה של המשתמשים חי (הצעות, לקוחות, הגדרות).
> **הסדר:** קודם גיט (נקודת התחלה נקייה ל-`App_v43`), אחר כך פרויקט Firebase אמיתי, ורק אז אבטחה ומנוי. גיט לא נותן מסד נתונים — זה בא אחריו.

1. **Git:** `App_v43.jsx` ל-repo. נקודת התחלה נקייה, ניהול גרסאות אמיתי.
2. **Firebase אמיתי משלך:** ליצור פרויקט Firebase, להחליף את ההסתמכות על `__firebase_config`/`__initial_auth_token` המוזרקים ב-config אמיתי. אימות אמיתי (Email / Google) במקום אנונימי.
2. **כללי אבטחה (Firestore Security Rules):** קריטי לפני פרודקשן — כל משתמש קורא/כותב רק תחת `users/{uid}` שלו. כיום אין הגנה אמיתית מעבר לטוקן של Studio.
3. **מנוי/תשלום (Paywall):** Stripe / RevenueCat / ספק ישראלי. מסך הרשמה, ניהול מנוי, חסימת פיצ'רים לפי מנוי.
4. **החלטת build:** לשקול מעבר מ-single-file ל-Vite + Tailwind build + פיצול לרכיבים — לתחזוקתיות. יתרון: ניתן לבדיקה/בנייה/CI. חיסרון: שובר את עקרון ה-no-build ודורש מיגרציה. החלטה אסטרטגית.
5. **גיבוי ושחזור אמיתיים, error monitoring (Sentry), אנליטיקס.**
6. **רב-לשוני** (כשתוכח דרישה).
7. **בדיקות אוטומטיות** למנועי הליבה (תמחור, תשלומים, חלוקת רווח) — כבר יש מתודולוגיית dry-run לבסס עליה.
